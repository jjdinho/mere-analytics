- **Resolve the `mere_readonly` source-of-truth question at start of step 10** (deploy / `users.xml`). Today the app creates and re-asserts `mere_readonly` on every boot (`internal/clickhouse/bootstrap.go:ProvisionReadonlyUser`) because no `users.xml` exists yet. When step 10 ships `config/deploy/clickhouse/users.xml`, *do whichever option is safer* — explicitly evaluated, not implicit. The two paths:
  - **(A) `users.xml` defines admin only; app remains sole source of truth for readonly.** Removes the two-sources-of-truth risk. Keeps drift correction (someone running `GRANT` manually gets reverted on next deploy). Keeps rotation convergence (changing `CLICKHOUSE_READONLY_PASSWORD` in `.kamal/secrets` takes effect on the next boot's `ALTER`, no CH config-reload race). One audit point: read the bootstrap function.
  - **(B) `users.xml` defines both users; app keeps the idempotent provisioning as belt-and-suspenders.** Matches `docs/plan.md`'s deploy section as drafted. But: if `users.xml` and `.kamal/secrets`/env diverge during a botched rotation, the app's `ALTER` wins on the next boot — which may *or may not* be what the operator wants. Two places to audit when state looks wrong.
  - **(C) `users.xml` defines both; remove the app-side provisioning.** Cleanest separation but gives up drift correction and rotation convergence — exactly the safety net `ProvisionReadonlyUser` exists for.

  Default recommendation today: **(A)**. Reason: the failure mode of "two sources of truth disagree" is sneaky (silent for the app, surprises the operator later) whereas the failure mode of "app is the only place that creates readonly" is loud and obvious (read one file to see what's true). Re-evaluate at step 10 with whatever new context exists then. Effort: S (XML edit + maybe deleting the `CREATE USER` line from `ProvisionReadonlyUser`). Priority: P1 for step 10.
